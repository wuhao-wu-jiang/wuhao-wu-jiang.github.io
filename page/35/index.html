Quake Heap

Posted on 2020-04-02 Edited on 2020-05-04

Quake Heap [1] is a min heap (or max heap) that supports the following operations:

Operation Amortized Time
insert(x) O(1)O(1)O(1)
decrease-key(x,k) O(1)O(1)O(1)
delete-min() O(log⁡n)O(\log n)O(logn)

with O(n)O(n)O(n) space. For comparison based min heap, either insert(x)insert(x)insert(x) or delete-min() takes O(log⁡n)O(\log n)O(logn) amortized time. Otherwise, we would have an o(nlog⁡n)o(n \log n)o(nlogn) comparison based sorting algorithm.

In this blog we walk through its construction step by step and illustrate the philosophy behind. Some of the ideas also appear in other designs of min heaps.

Insertion

For insertion we are going to be lazy. Consider inserting the elements 1, 2, 3, 4, 5 into the heap. For each insertion, we just create a singleton node while maintaining a pointer (the “min-pointer”) to the minimum node. The picture below shows the heap we obtain after five insertions. Each insertion has time complexity O(1)O(1)O(1).

Deletion and Binomial Heap

It would be easier to consider the case delete-min() without decrease-key(x) first. Now we try to delete the minimum element from heap. The first step, of course, is to remove the element 111, after which we get the following heap.

Here comes the problem. The “min-pointer” is removed along with the deletion. To maintain it, it seems that we need to loop over the entire list to update the min, which takes O(n)O(n)O(n) time. If we do nothing else than merely scanning all elements, we need another O(n)O(n)O(n)-scanning for the next deletion. We hope that the comparison performed for the current delete-min operation can reduce the ones we need to perform in the future.

The idea is to merge the elements into trees. Intuitively, the trees “remember” the comparison performed before and save us from comparing the same pairs again.

Various tree structure can be used to record the comparison and this may result in the design of different data structure. The one picked by Quake heap is the tournament tree. It only compares and merges (linking) two trees of the same height. A new node is created as the root of the two trees. The new node’s value equals to the smaller one of the roots of the two trees.

For example, in the following example, both the node 4 and 5 are viewed as trees with height one. We can compare and link the two trees. A new node with value 4 (the smaller value between 4 and 5) is created as the new root of the merged tree.

We can continue to link node 3 and 2.

The linking is performed recursively. Noting the the current two trees have the same height, we link them. We need to also update the “min-pointer” during the linking.

The construction takes O(n)O(n)O(n) time. But we can charge the cost of these link operations into the creation of the trees’ roots before. We have performed 3 link operations for the four nodes each being the root of a singleton tree. On average, we pay O(1)O(1)O(1) per operation. To make the amortized argument more vivid, we can think that conceptually we an additional coin for each singleton tree creation. Therefore, before linking, we have four coins as follows:

Merging node 5 and 4 has actual cost O(1)O(1)O(1) and creates a new tree root. We use the coin of node 5 to pay the actual cost. Element 4 is the new root and it still has a coin. By using the coin idea, we charge (or averaging) the cost of this link operation to the cost of the creation of node 5.

Merging node 3 and 2 uses one coin. We use the coin reserved by node 3. After this, the elements 4 and 2 (the elements that are roots of some trees) still have a coin.

Merging the two trees left uses one coin. We use the coin reserved by node 4. After this, the element 2 is still a root and still has a coin.

In general, we may need to pay more than O(1)O(1)O(1) for delete-min(). Consider we would like to perform the operation for the following.

We need to remove all nodes on a root-to-leaf path. In this example it is the path that contains element 2.

After the removal, some children of the nodes in this root-to-leaf path are now exposed as tree roots. In this example, node 4 and 3 are tree roots. We need to reserve two coins node 4 and 3. The coins are reserved for potential linking performed in the future.

Finally we need to compare and link any two trees of the same height, in a recursive manner. This part is paid by the coins reserved for the tree roots.

To analyze the amortized running time of delete-min(), we define the height of the tree as HHH. We need to remove HHH nodes from the root-to-leaf path, expose max⁡{H−1,0}\max \{H - 1, 0 \}max{H−1,0} node as tree roots and reserve coins for them, followed by recursively linking two trees with the same height. The final step is “free” and covered by the coins reserved. Therefore the amortized cost is H+max⁡{H−1,0}=max⁡{2H−1,H}≤2HH + \max \{H - 1, 0\} = \max \{ 2H - 1, H \} \le 2HH+max{H−1,0}=max{2H−1,H}≤2H.

Key Observation:

To bound the time of delete-min() to O(log⁡n)O(\log n)O(logn), it is critical to control the size of HHH.

To analyze how large HHH could be, we study how it grow. Initially, a tree with H=1H = 1H=1 contains only 1 key. When we link two trees with H=1H = 1H=1, the new tree has height H=2H = 2H=2 and 222 keys (although it has 333 nodes). In general, a tree with height H=hH = hH=h contains 2h−12^{h - 1}2h−1 keys and this can be easily proved by induction.

If there is no decrease-key operation, the tree with height hhh has 2h−12^{h - 1}2h−1 keys and 2h−12^h - 12h−1 nodes. As there are at most nnn keys, the maximum possible height, denoted as HHH, is bounded by log⁡n\log nlogn.

The heap we have constructed so far has a special name called “Binomial Heap”.

Decrease-Key

If there is only insert() and decrease-key() operation, min heap is easy to implemented. We can use a linked list / dynamic array, while maintaining the min pointer / min index.

If we need to incorporate delete-min() operation, the decrease-key(x) itself is still easy, while the former requires significant modification. For decrease-key(x), we adopt a lazy strategy. If after decreasing, xxx’s value is still smaller than its parent (or xxx is the root of the tree and has no parent), then everything is fine. For example, if we decrease 444 to 3.53.53.5 (the node 4 has parent 2), then we do not need to modify the tree after the operation.

On the other hand, if we decrease 4 to 1, then its value is smaller than its parent 2.

We need to detach the highest node containing value 4 from its parent node (in the meanwhile we maintain the “min-pointer”).

The only concern with this operation is the height of a heap. In the previous, the tree on the right hand side has height 333 but contains only two keys. For a tree with nnn keys, it could happens that it height is ω(log⁡n)\omega(\log n)ω(logn). Indeed, it could be as large as O(n)O(n)O(n). This will invalidate the strategy we propose in the last section, which requires the height to be O(log⁡n)O(\log n)O(logn), to achieve the O(log⁡n)O(\log n)O(logn) amortized complexity for delete-min().

There are various ways to fix this, the idea of which will give rise to Pairing heap [Pairing heap achieve O(log⁡n)O(\log n)O(logn) amortized bound for deletion but requires O(log⁡n)O(\log n)O(logn) amortized time for both insertion and decrease-key(x). Its design philosophy is closely related to Splay tree.], Fibonacci heap, etc. Here we discuss about the solution of Quake heap.

First, Quake heap records the number of nodes at each height explicitly, denoted as n1,n2,...,nHn_1, n_2, ..., n_Hn1​,n2​,...,nH​, where HHH is the maximum possible height. Further, it specifies a parameter α∈(0,5,1)\alpha \in (0, 5, 1)α∈(0,5,1) and maintains the invariant that ni+1≤α⋅nin_{i + 1} \le \alpha \cdot n_ini+1​≤α⋅ni​ for each i≥1i \ge 1i≥1. This guarantees that H=O(log⁡n)H = O(\log n)H=O(logn).

In the following discussion, we take α=0.75\alpha = 0.75α=0.75.

Let’s discuss how the maintain the invariant ni+1≤αnin_{i + 1} \le \alpha n_ini+1​≤αni​ for all i≥1i \ge 1i≥1 for insert(x), decrease-key(x) and delete-min() operation.

  1. For the operation insert(x), it create a singleton node at the bottom layer and increases n1n_1n1​ by 1. If the invariant holds before insertion, it holds afterward.

  2. For decrease, it will not change the number of nodes at each height. The following example shows the case where the children whose key is decreased is detached from its parent.

  3. Delete-min is the most difficult part. To maintain the invariant, now delete-min() consists of four part:

    • Delete the min key (and all nodes that contain the min key). As before, this step may expose some children of the deleted nodes as new tree roots.
    • Recursively link all trees of the same height. After linking, scan all trees to maintain the min pointer.
    • Quake: if there is a level iii such that ni+1>αnin_{i + 1} > \alpha n_ini+1​>αni​, remove all nodes with level greater than iii.

It is left to analyze the running time step by step. We use a potential function for analysis. By previous section, the potential should contain the number of trees. Further, by the last step of delete-min operation, it is suggested to includes the number of nodes in the potential function, to cover the cost of node removal. Inspired by this, our first design is given by

Φ=#nodes+2⋅#trees\Phi = \# \text{nodes} + 2 \cdot \# \text{trees} Φ=#nodes+2⋅#trees

We analyze the time for each step of delete-min()

  1. Before delete-min(), the invariant holds and therefore the maximum height H=O(log⁡n)H = O(\log n)H=O(logn). Deleting the min-key and its associated nodes take actual time O(log⁡n)O(\log n)O(logn). As for Φ\PhiΦ, the number of nodes decreases (indeed, this decrease will cover the removal of the nodes containing the min key) and the number of trees increases by at most O(log⁡n)O(\log n)O(logn). Therefore, the amortized cost for the first step is O(log⁡n)O(\log n)O(logn).

  2. Linking two trees of the same height has actual cost 111. It creates a new node and decreases the number of trees by 1. Therefore, ΔΦ=−1\Delta \Phi = -1ΔΦ=−1. Hence the amortized cost is 0. This also explains why we need an coefficient 2 associated with #trees in the potential function. It is left to analyze the cost of scanning all tree. After linking, all trees are of different height. It suffices to show that the maximum height is O(log⁡n)O(\log n)O(logn) to bound the scanning cost to O(log⁡n)O(\log n)O(logn). Before linking H=O(log⁡n)H = O(\log n)H=O(logn). As there are only nnn keys, there are at most nnn trees with height HHH before and during the linking procedure. Linking nnn trees of the same height results in a new tree with height log⁡n\log nlogn. Therefore, after linking, HHH increases by at most log⁡n\log nlogn and H=O(log⁡n)H = O(\log n)H=O(logn) still holds. Therefore, the amortized cost for the second step is O(log⁡n)O(\log n)O(logn).

  3. Suppose in this step we delete kkk nodes. The actual cost is kkk. However, the number of nodes decreases by kkk. It seems that the amortized cost is 0 for this step. But it isn’t. We have increased the number of tree. Denote di+1d_{i + 1}di+1​ the number of nodes at level i+1i + 1i+1 with two children before removal and si+1s_{i + 1}si+1​ the number of nodes with only one children. We have ni+1=di+1+si+1n_{i + 1} = d_{i + 1} + s_{i + 1}ni+1​=di+1​+si+1​. On the other hand, we know ni≥2di+1+si+1n_i \ge 2 d_{i + 1} + s_{i + 1}ni​≥2di+1​+si+1​. By the fact that ni+1>αnin_{i + 1} > \alpha n_ini+1​>αni​, we know that

    si+1=2(di+1+si+1)−(2di+1+si+1)≥2ni+1−ni≥(2α−1)ni\begin{aligned} s_{i + 1} &= 2(d_{i + 1} + s_{i + 1}) - (2 d_{i + 1} + s_{i + 1})\\ &\ge 2 n_{i + 1} - n_i \\ &\ge (2 \alpha - 1) n_i \end{aligned} si+1​​=2(di+1​+si+1​)−(2di+1​+si+1​)≥2ni+1​−ni​≥(2α−1)ni​​

    as we create at most nin_ini​ new trees, we can charge the creation of new trees into si+1s_{i + 1}si+1​ and modify the potential as

    Φ=#nodes+2⋅#trees+22α−1#nodes with one child\Phi = \# \text{nodes} + 2 \cdot \# \text{trees} + \frac{2}{2\alpha - 1} \# \text{nodes with one child} Φ=#nodes+2⋅#trees+2α−12​#nodes with one child

    Now step 3 has amortized cost 0. For other steps, this modification will increase the cost of decrease-key(x) by at most O(1)O(1)O(1).







Caveat: we can’t swap the order of step 2 and step 3, since linking tree may create violation to the invariant and we need step 3 to fix this.
