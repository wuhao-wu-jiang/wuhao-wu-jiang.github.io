Splay Tree

Posted on 2020-05-26 Edited on 2022-01-28

Splay Tree [1] is a self-binary search tree that achieves O(log⁡n)O(\log n)O(logn) amortized time complexity for each operation.
Unlike other balanced binary search trees, it does not require to maintain additional information to keep balance.
Further, it has good property that more recently accessed nodes are more easily retrieved and achieves certain optimality even for an unknown sequence of operations.

Basic Operations

The philosophy of splay tree is very simple: just bring the last accessed node xxx to the root of the tree, via a sequence of rotations.
This brings the most recently visited nodes close to the root.

[Remark: For insertion, the last accessed node is the node inserted.
For a successful search operation, the last accessed node is the node found.
For an unsuccessful search/deletion, it is the predecessor or successor of the search node / deleted node.]

There is more than one way to rotate a node xxx to the root.
For example, the simplest one rotates xxx with its parent repeatedly until it becomes the root.
However, we can easily find counter examples for this naïve approach.
The rotations need to be carefully designed and are called splays.
At each step the node being splayed is brought up at most 2 levels. It can be divided into three categories.

Call xxx the node being splayed, yyy its parent and zzz its grandparent (if they exist).

  1. Root case: xxx is the root. Do nothing.

Zig

  1. The Zig case: yyy is the root. We just perform rotate(x,y)rotate(x, y)rotate(x,y).

    • See the following figure.

  2. Otherwise, xxx has a grandparent zzz. It is further divided into two categories:

Zig-Zig

  • Zig-Zig case: both xxx and yyy are left children or right children. We first perform rotate(y,z)rotate(y, z)rotate(y,z) then rotate(x,y)rotate(x, y)rotate(x,y).

Zig-Zag

  • Zig-Zag case: xxx and yyy are different children of their parents, i.e., xxx is the left child and yyy is the right child, or vice versa. We first perform rotate(x,y)rotate(x, y)rotate(x,y) then rotate(y,z)rotate(y, z)rotate(y,z).

The splay is performed repeatedly on xxx until it reaches the root. It is also possible to perform equivalent operations in an top-down approach [1].

Amortized Analysis

There is no explicit control over the height of the tree.
It is possible that the accessed node has depth Ω(log⁡n)\Omega(\log n)Ω(logn).
The intuition is that, however, the imbalanced situation is not easy to create and results from a sequence of previous operations.
The number such operations can not be too small to create the imbalance.
We can charge the cost of accessing the node with long depth to these previous operations.
Intuitively, each such operation is required to reserve some additional “energy” for the amount of imbalanced it creates, which is used later to fix the imbalance.
When a long path is encountered, there should be enough “energy” to cover the search on the path, and the subsequent splay costs.

It is left to give a quantitative analysis for keeping the “energy” and for “using” the “energy”.
We need a few more definitions for a node xxx.

  1. T(x):T(x):T(x): the subtree rooted at xxx.

  2. w(x):w(x):w(x): the weight associated with node xxx. This weight is only used for analysis and is not kept explicitly by splay tree.

  3. s(x)=∑y∈T(x)wy:s(x)=\sum_{y \in T(x)} w_y:s(x)=∑y∈T(x)​wy​: the sum of weights of nodes in T(x)T(x)T(x), which is referred to as the size of xxx.

  4. r(x)≐log⁡s(x)r(x) \doteq \log s(x)r(x)≐logs(x), the rank of xxx.

Next, we analyze the amortized costs of basic BST operations.
Throughout the analysis we assume that w(x)=1w(x) = 1w(x)=1.

Search

Whether the search is successful or not, we splay the last accessed node to the root and charge the cost of going down the tree to the splay-operations.
Therefore, the amortized cost is O(log⁡n)O(\log n)O(logn).

Remark: in case of unsuccessful search, we find either the predecessor or successor of the search key.

Insertion

Inserting a node xxx is the same as that in a BST.
Then we splay node xxx to the root.
We charge the cost of going down the tree by the cost of splaying xxx, which has amortized cost O(log⁡n)O(\log n)O(logn).
It is left to analyze the change of Φ\PhiΦ.

Deletion

As in a BST, deletion is a little bit tricky.
Let xxx be the node to delete and yyy be its parent (if exists). There are 3 cases

  • If xxx is a leaf node, then we just delete it and splay yyy to the root.

  • If xxx has only one child, we replace xxx with its child as the child of yyy. Then we delete xxx and splay yyy to the root.

The final case uses the first two cases as a sub-routine. We first discuss the amortized cost of the first two cases. The cost of going down to xxx can be charged to the cost of splaying yyy to the root, which has amortized cost O(log⁡n)O(\log n)O(logn). Deleting the node decreases the potential function, which will only lower the overall amortized cost. Therefore, the amortized cost of deletion is bounded by O(log⁡n)O(\log n)O(logn).

  • If xxx has two children, we find the predecessor node pred(x)pred(x)pred(x) of xxx, which is the node with largest key in xxx’s left sub-tree. Note that pred(x)pred(x)pred(x) has at most one child. Then we delete the node pred(x)pred(x)pred(x). This reduces to case 1 or case 2. Finally we replace the key(x)key(x)key(x) with key(pred(x))key(pred(x))key(pred(x)).
