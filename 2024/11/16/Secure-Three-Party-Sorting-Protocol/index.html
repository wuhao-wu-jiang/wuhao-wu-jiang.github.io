Secure Three Party Sorting Protocol

Posted on 2024-11-16 Edited on 2024-12-01

Let \(p\) be some prime number, \(\mathbb{F}_p\) be a finite field of order \(p\), and \(\vec{x} = (x_1, \ldots, x_n) \in \mathbb{F}_p^n\). Let \(\ell \doteq \lceil -1 + \log_2 (p + 1) \rceil.\) For each \(x_i\), denote its binary representation as \(\mathbb{B}in(x_i) = (b_{i, \ell}, b_{i, \ell - 1}, \ldots, b_{i, 0})\), s.t., \[ x_i \doteq \sum_{j \in 0}^\ell b_{i, j} \cdot 2^j. \]

Permutation. A permutation \(\pi : [n] \leftrightarrow [n]\) is a bijection between \([n]\) and \([n]\).

We can apply a permutation \(\pi\) to a vector \(\vec{x}\), to obtain a new vector \(\pi(\vec{x}) \doteq (x_{\pi(1)}, \ldots, x_{\pi(n)}).\)

We present an alternative view of the radix sort.

Algorithm (Radix Sort)
1. For \(j \leftarrow 0\) to \(\ell\) do
2. \(\quad\) Obtain a permutation \(\pi\), s.t., \(\pi(\vec{x})\) is a stable sort of \(\vec{x}\) according to the \(j^{(th)}\) bit. 3. \(\quad\) Apply the permutation \(\vec{x} \leftarrow \pi(\vec{x})\)

It remains to convert this into a secure sorting protocol, for three parties. We need to show that at each iteration,

  1. A permutation \(\pi\) can be generated in a secure manner, and
  2. The permutation \(\pi\) can be applied in a secure manner.

Generating Secure Permutation for Stable Sort

In this section, we discuss the protocol for stably sorting \(\vec{x}\) according to the \(j^{(\text{th})}\) bits. This reduces to designing a stable sorting protocol for an array \((a_1, \ldots, a_n)\) of binary bits.

The key observation is that, for each \(a_i\), we can explicitly compute its position, i.e., \(\pi(i)\). In particular, for each \(i\), define \[ \begin{aligned} s_0(i) &\doteq \{ j \in [i - 1] : a_j = 0 \}, \\ s_1(i) &\doteq \{ j \in [i - 1] : a_j = 1 \}. \end{aligned} \] i.e., \(s_0(i)\) and \(s_1(i)\) are the number of zeros and ones before \(a_i\), respectively. If \(a_i = 0\), after sorting, it should be place in the \(s_i(0)\)-th position, and if \(a_i = 1\), it should be placed in the \(\big( s_i(0) + s_i(1) \big)\)-th position.

The above description actually describe the inverse of the permutation \(\pi\), which stably sorts the input array \((a_1, \ldots, a_n)\), which is given by the following formula: \[ \pi^{-1} (i) = s_0(i) + s_1(i) \cdot a_i, \quad \forall i \in [n]. \]

Algorithm.
1. \(\llbracket s_0 \rrbracket \leftarrow 0\)
2. \(\llbracket s_1 \rrbracket \leftarrow 0\)
3. For \(i \in [n]\) do
4. \(\quad\) \(\llbracket s_0 \rrbracket \leftarrow \llbracket s_0 \rrbracket + (1 - \llbracket a_i \rrbracket)\)
5. \(\quad\) \(\llbracket s_1 \rrbracket \leftarrow \llbracket s_1 \rrbracket + \llbracket a_i \rrbracket\)
6. \(\quad\) \(\llbracket \pi^{-1}(i) \rrbracket \leftarrow \llbracket s_0 \rrbracket + \llbracket s_1 \rrbracket \cdot \llbracket a_i \rrbracket\)
7. return \(\llbracket \pi^{-1} \rrbracket \leftarrow \big( \llbracket \pi^{-1} (1) \rrbracket, \ldots, \llbracket \pi^{-1} (n) \rrbracket \big)\).

The above protocol involves only addition and multiplication of secret shares.

Applying a Permutation

The above protocol compute \(\pi^{-1}\) as secret shares. The following question is, how can we apply it to \((a_1, \ldots, a_n)\), to obtain a sorted sequence. We don't want to recover \(\pi^{-1}\) from its secret shares, as \(\pi^{-1}\) itself contains information about \((a_1, \ldots, a_n)\).

We can apply a special trick to get around this. Assume that \(\sigma\) is a random permutation. Consider applying it to both \(\pi^{-1}\), and \((a_1, \ldots, a_n)\): \[ \begin{array}{ccc} \sigma(\pi^{-1}) &= \quad (& \pi^{-1}(\sigma(1)), &\ldots, &\, \pi^{-1}(\sigma(n)) &), \\ \sigma(a_1, \ldots, a_n) &= \quad (& a_{\sigma(1)}, &\ldots, &\, a_{\sigma_n} &). \end{array} \] Observe that, since \(\pi^{-1}(i)\) specifies the position of \(a_i\), and \(\sigma\) is a bijection, \(\pi^{-1}(\sigma(i))\) specifies exactly the position of \(a_{\sigma(i)}\) for each \(i \in [n]\).

Local Additive

Assume that there are 3 parties, s.t., the party \(i\) holds a secret share \(\llbracket a \rrbracket_i\) for \(i \in [3]\).

Then there is a protocol, in which party \(1\) generates a secret \(b\), and party \(2\) generates a secrete \(c\), s.t., \(a = b + c\), while both parties do not learn about \(a\).

Assume that \[ \llbracket a \rrbracket_1 = g(1) = a + t \cdot 1, \quad \llbracket a \rrbracket_2 = g(2) = a + t \cdot 2. \]

One possible protocol is as follows:

  • Party \(1\): computes \[ b = 2 \cdot \llbracket a \rrbracket_1 = 2 \cdot a + t \cdot 2 \]
  • Party \(2\): on receiving \(b\), compute \[ c = - \llbracket a \rrbracket_2 = - 2 \cdot a - t \cdot 2 \]
