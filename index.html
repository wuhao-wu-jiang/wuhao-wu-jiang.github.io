Turing Machine Decidability

Posted on 2025-04-09 Edited on 2025-04-10

We are interested in whether a problem is algorithmically unsolvable. Knowing when a problem is algorithmically unsolvable is useful because then you realize that the problem must be simplified or altered before you can find an algorithmic solution.

Decidable Languages

Regular Languages

Theorem. Let
\(A_{DFA} = \{ \langle B, w \rangle \mid B \text{ is a DFA that accepts input string } w \}\),
\(A_{NFA} = \{ \langle B, w \rangle \mid B \text{ is an NFA that accepts input string } w \}\), and
\(A_{REX} = \{ \langle R, w \rangle \mid R \text{ is a regular expression that generates string } w \}\).
Then \(A_{DFA}\), \(A_{NFA}\), and \(A_{REX}\) are decidable languages.

Proof Hint. The proof involves constructing Turing machines that simulate the behavior of a DFA, NFA, or a regular expression on the given input—similar to how such simulations are implemented in standard programming languages.

Theorem. Let \(E_{DFA} = \{ \langle A \rangle \mid A \text{ is a DFA and } L(A) = \emptyset \}\).
Then \(E_{DFA}\) is a decidable language.

Proof Hint. Design a Turing machine that checks whether there exists a path from the start state to any accept state.

Theorem. Let \(EQ_{DFA} = \{ \langle A, B \rangle \mid A \text{ and } B \text{ are DFAs and } L(A) = L(B) \}\).
Then \(EQ_{DFA}\) is a decidable language.

Proof Hint. Test whether both \(L(A) \cap \overline{L(B)}\) and \(L(B) \cap \overline{L(A)}\) are empty (since regular languages are closed under complement and intersection, both resulting languages are regular). If both are empty, then \(L(A) = L(B)\).

Context-Free Languages

Theorem. Let \(A_{CFG} = \{ \langle G, w \rangle \mid G \text{ is a CFG that generates string } w \}\).
Then \(A_{CFG}\) is a decidable language.

Theorem. Let \(E_{CFG} = \{ \langle G \rangle \mid G \text{ is a CFG and } L(G) = \emptyset \}\).
Then \(E_{CFG}\) is a decidable language.

Theorem. Every context-free language is decidable.

Proof Hint. * Simulate a parsing algorithm for CFGs using a Turing machine.
* One possible parsing algorithm: Convert the CFG to Chomsky Normal Form (CNF) using the following steps:
- Add a new start variable \(S_0\) and the rule \(S_0 \rightarrow S\)
- Eliminate all \(\epsilon\)-productions of the form \(A \rightarrow \epsilon\)
- Eliminate all unit productions (rules of the form \(A \rightarrow B\))
- Convert all remaining rules into CNF

If $G$ is in CNF, any derivation of a string $w \in L(G)$ has exactly $2|w| - 1$ steps, which bounds the parsing time.

Undecidable Language

Theorem. Let \(A_{TM} = \{ \langle M, w \rangle \mid M \text{ is a TM and } M \text{ accepts } w \}\).
Then \(A_{TM}\) is undecidable.

The Turing machine \(U\) is interesting in its own right.

Universal Turing Machine
$U = $ “On input \(\langle M, w \rangle\), where \(M\) is a TM and \(w\) is a string:

  1. Simulate \(M\) on input \(w\).
  2. If \(M\) ever enters its accept state, accept; if \(M\) ever enters its reject state, reject.”

Theorem. Some languages are not Turing-recognizable.

Proof Hint.
Assume that a TM \(H\) decides \(A_{TM}\). Use \(H\) to build a TM \(D\) that takes input \(\langle B \rangle\) and accepts it exactly when \(B\) does not accept \(\langle B \rangle\). Finally, run \(D\) on itself. The contradiction arises as follows:

  • \(H\) accepts \(\langle M, w \rangle\) exactly when \(M\) accepts \(w\).
  • \(D\) rejects \(\langle B \rangle\) exactly when \(B\) accepts \(\langle B \rangle\).
  • \(D\) rejects \(\langle D \rangle\) exactly when \(D\) accepts \(\langle D \rangle\) — a contradiction.

We say that a language is co-Turing-recognizable if it is the complement of a Turing-recognizable language.

Theorem. A language is decidable iff it is both Turing-recognizable and co-Turing-recognizable.

Corollary. \(A_{TM}\) is not co-Turing-recognizable.
Proof. We know \(A_{TM}\) is Turing-recognizable.
If \(A_{TM}\) were also co-Turing-recognizable, then \(A_{TM}\) would be decidable (by the theorem above).
But we know \(A_{TM}\) is undecidable, so \(A_{TM}\) cannot be co-Turing-recognizable.

BGW Protocol

Posted on 2024-11-09 Edited on 2025-08-27

The BGW is a secure multiparty computation protocol for polynomials over a finite field.

Model

Let \(p\) be a prime number, and let \(\mathbb{F}_p\) be a finite field of order \(p\).

In the setting, there are \(n\) parties, where the party \(i\) holds data \(b_{i, 0} \in \mathbb{F}_p\). The objective is to compute a polynomial \(f (x_1, \ldots, x_n)\) over \(\mathbb{F}_p\) in a privacy-preserving manner.

Privacy Guarantee. We assume all parties are curious but honest. A protocol is considered \(t\)-private, if any set of up to \(t\) players cannot learn more from protocol's outcome than they could jointly infer from their own private inputs and outputs.

Protocol

We need the following protocol as a primitive for sharing information between the parties.

Shamir Secret Sharing Scheme \(\mathcal{S}\)
Party \(i \in [n] :\)
1. Sample independent random primes \(b_{i, 1}, \ldots, b_{i, t} \in \mathbb{F}_p\).
2. Let \(\vec{b}_i \doteq (b_{i, 0}, b_{i, 1}, \ldots, b_{i, t}).\)
3. Let \(g_{\vec{b}_i} (x) \doteq b_{i, 0} + b_{i, 1} \cdot x + \ldots + b_{i, t} \cdot x^t\).
4. For each user \(j \in [n]\):
5. \(\quad\) Send \(g_{\vec{b}_i} (j)\) to user \(j\)

We call \(g_{\vec{b}_i} (j)\) a share of the secret \(b_{i, 0}\).

Privacy Guarantee. It is easy to verify that \(\mathcal{S}\) is \(t\)-private.

The key result we would like to establish is that a polynomial \(f(x_1, \ldots, x_n)\) can also be computed in a \(t\)-private manner.

Since a polynomial can be constructed through additions, scalar multiplication, and the multiplication of (smaller) polynomials, it is sufficient to demonstrate how these operations can be performed in a \(t\)-private manner. We begin by discussing the addition protocol. Protocols for subtraction and scalar multiplication can be derived using similar techniques. Finally, we present the multiplication protocol.

Addition

We start with a simple protocol, which tries to add up \(b_{k, 0}\) and \(b_{\ell, 0}\), for some parties \(k, \ell \in [n]\).

Adding \(b_{k, 0}\) and \(b_{\ell, 0}\):

Party \(k\):

  • Invoke \(\mathcal{S}\) to send \(g_{\vec{b}_k}(j)\) to each \(j \in [n]\).

Party \(\ell\):

  • Invoke \(\mathcal{S}\) to send \(g_{\vec{b}_\ell} (j)\) to each \(j \in [n]\).

Party \(i \in [n]\):

  • On receiving \(g_{\vec{b}_k}(i)\) and \(g_{\vec{b}_\ell}(i)\), compute \(g_{\vec{b}_k}(i) + g_{\vec{b}_\ell}(i)\).

The key observation is that the following relation holds for polynomials: \[ \begin{aligned} g_{\vec{b}_k}(x) + g_{\vec{b}_\ell}(x) &= g_{\vec{b}_k + \vec{b}_\ell}(x) \\ &= (b_{k, 0} + b_{\ell, 0}) + (b_{k, 1} + b_{\ell, 1}) \cdot x + \ldots + (b_{k, t} + b_{\ell, t}) \cdot x^t. \end{aligned} \]

Composition in the View of a Virtual Party

Thus, the addition operation can be viewed as a virtual party \(v\) holding data \(b_{k, 0} + b_{\ell, 0}\), which is then shared with all parties \(i \in [n]\) via the Shamir Secret Sharing Scheme. In this scheme, the polynomial coefficients are \(\vec{b}_k + \vec{b}_\ell\). Consequently, each party \(i\) receives a share of the secret, denoted as \(g_{\vec{b}_k + \vec{b}_\ell}(i)\).

Virtual Party \(v\): * Invoke \(\mathcal{S}\) to send \(g_{\vec{b}_k + \vec{b}_\ell}(j)\) to each \(j \in [n]\).

This alternative view facilitates a clearer understanding of composition.

For example, suppose we want to compute \(b_{k, 0} + b_{\ell, 0} + b_{m, 0}\) for some parties \(k, \ell, m \in [n]\). First, we can apply the protocol described above to compute \(b_{k, 0} + b_{\ell, 0}\).

Then, we consider two parties: a virtual party \(v\) holding data \(b_{k, 0} + b_{\ell, 0}\), and party \(m\) with data \(b_{m, 0}\). We can then invoke the addition protocol again to compute \(b_{k, 0} + b_{\ell, 0} + b_{m, 0}\). Importantly, communication with the virtual party is not actually required, as after the first protocol invocation, each party \(i\) already possesses the secret share \(g_{\vec{b}_k + \vec{b}_\ell}(j)\).

Privacy Guarantee. After the addition protocol, each party \(i\) receives a share of the secret \(b_{k, 0} + b_{\ell, 0}\), represented by \(g_{\vec{b}_k + \vec{b}_\ell}(i)\). Since \(g_{\vec{b}_k + \vec{b}_\ell}(x)\) is a random polynomial of degree \(t\), the output of the addition protocol remains \(t\)-private.

Multiplication

The multiplication is more intricate. Note that: \[ \begin{aligned} g_{\vec{b}_k}(x) \cdot g_{\vec{b}_\ell}(x) &= g_{\vec{b}_k \times \vec{b}_\ell}(x) \\ &= \left( b_{k, 0} \cdot b_{\ell, 0} \right) + \left(\sum_{i, j \ge 0, \, i + j = 1} b_{k, i} \cdot b_{\ell, j}\right) \cdot x + \cdots + \left(\sum_{i, j \ge 0, \, i + j = 2 t} b_{k, i} \cdot b_{\ell, j}\right) \cdot x^{2 t}. \end{aligned} \]

There are two problems with this polynomial:

  1. The coefficients of this polynomial are not independent.
  2. The polynomial has degree \(2t\), instead of \(t\).

Hence the previous security analysis of the Shamir Secret Sharing Scheme does not hold.

Rerandomization

To fix the first problem, each party \(i\) can generate a random polynomial \[ g_i (x) = 0 + c_{i, 1} \cdot x + c_{i, 2} \cdot x^2 + \cdots + c_{i, 2t} \cdot x^{2t}, \] then they sends the shares \(g_i(1), \ldots, g_i(n)\) to other parties.

After sending and receiving new shares, each party \(i\) then computes \(g_{\vec{b}_k \times \vec{b}_\ell}(i) + \sum_{j \in [n]} g_j(i)\). The polynomial of interest is given by

\[ \begin{aligned} g_{\vec{b}_k \times \vec{b}_\ell}(x) + \sum_{i \in [n]} g_i (x) &= \left( b_{k, 0} \cdot b_{\ell, 0} \right) + \left(\sum_{i, j \ge 0, \, i + j = 1} b_{k, i} \cdot b_{\ell, j} + \sum_{i \in [n]} c_{i, 1} \right) \cdot x + \cdots + \left(\sum_{i, j \ge 0, \, i + j = 2 t} b_{k, i} \cdot b_{\ell, j} + \sum_{i \in [n]} c_{i, 2t} \right) \cdot x^{2 t}. \end{aligned} \]

Note this is a random polynomial, in the sense that the coefficients of this polynomial are independent. Therefore, the outputs \(g_{\vec{b}_k \times \vec{b}_\ell}(i) + \sum_{j \in [n]} g_j(i), i \in [n]\) are \(2t\)-private.

Remark. It appears sufficient for one party \(i\) (if trusted) to generate a random polynomial \(g_i\) and share \(g_i(1), \ldots, g_i(n)\) with other parties. Essentially, this adds a random coefficient to each coefficient of \(g_{\vec{b}_k \times \vec{b}_\ell}(x)\).

Truncation

To simplify notation, let \[ h(x) \doteq c_0 + c_1 \cdot x + \ldots + c_{2t} \cdot x^{2t} \] be a shorthand for the polynomial \(g_{\vec{b}_k \times \vec{b}_\ell}(x) + \sum_{i \in [n]} g_i (x)\). We have shown that \(c_1, \ldots, c_n\) are random integers from \(\mathbb{F}_p\).

Our goal is to truncate its degree from \(2t\) to \(t\): \[ \bar{h}(x) \doteq c_0 + c_1 \cdot x + \ldots + c_{t} \cdot x^{t}, \] and for each party \(i \in [n]\), replaces its share \(h(i)\) by \(\bar{h}(i)\).

A naive approach is to reveal the secret shares of a set of \(2t\) clients, for example \(h(1), \ldots, h(2 t)\), so that we can recover the entire polynomial \(h(x)\) (and therefore the secret value \(h(0) = c_0\)).

Can we achieve this in a \(t\)-private manner? The answer is "Yes"! We are going to show that, there is a matrix \(T\), s.t., \[ \bar{h}_{1 : n} \doteq \begin{bmatrix} \bar{h}(1) \\ \bar{h}(2) \\ \vdots \\ \bar{h}(n) \end{bmatrix}, \quad h_{1 : n} = \begin{bmatrix} h(1) \\ h(2) \\ \vdots \\ h(n) \end{bmatrix}, \quad \bar{h}_{1 : n} = T \times h_{1 : n}. \]

Observe that, multiplying \(h_{1 : n}\) by \(T\) only involves addition and scalar multiplications of secret shares, which we have already shown can be performed in a \(t\)-private manner. It remain to explain what \(T\) is.

Define \[ V = \begin{bmatrix} 1, &1, &\cdots, &1 \\ 1, &2, &\cdots, &2^{n - 1} \\ &&\cdots\, \\ 1, &i, &\cdots, &i^{n - 1} \\ &&\cdots\, \\ 1, &n, &\cdots, &n^{n - 1} \end{bmatrix}, \quad \vec{c} = \begin{bmatrix} c_0 \\ c_1 \\ \vdots \\ c_{2t} \\ 0 \\ \vdots \\ 0 \end{bmatrix}, \quad \bar{c} = \begin{bmatrix} c_0 \\ c_1 \\ \vdots \\ c_{t} \\ 0 \\ \vdots \\ 0 \end{bmatrix}. \]

Then \(h_{1: n} = V \times \vec{c}\), and \(\bar{h}_{1: n} = V \times \bar{c}\). We also know that there exists a projection matrix \(P\), s.t., \(\bar{c} = P \vec{c}\). It follows that \[ \bar{h}_{1 : n} = V \times \bar{c} = V \times P \times \vec{c} = V \times P \times V^{-1} \times h_{1 : n}. \] Taking \(T = V \times P \times V^{-1}\) computes the proof.

Remark. If we directly compute \(V^{-1} \times h_{1 : n}\), we recover all coefficients of \(h(x)\). The key trick is to compute \((V \times P \times V^{-1}) \times h_{1 : n}\) instead, bypassing the explicit computation of \(V^{-1} \times h_{1 : n}\).
