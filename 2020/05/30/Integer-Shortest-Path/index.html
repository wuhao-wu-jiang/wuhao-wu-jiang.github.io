Integer Shortest Path

Posted on 2020-05-30 Edited on 2020-06-17

Single source shortest path is one of most popular the topics taught in a introductory algorithm course. Given a graph G=<V,E>G = \left< V, E \right>G=⟨V,E⟩ with nnn vertices and mmm edges, the famous Dijkstra’s algorithm has runtime O(mlog⁡n)O(m \log n)O(mlogn) by using a binary heap. When more advanced heap such such Fibonacci heap is deployed, the runtime can be improved to O(m+nlog⁡n)O(m + n \log n)O(m+nlogn). Most introductory courses are satisfied with the above solutions without going further.

Here we explore the topic by considering a special case where all edge weights are non-negative integers within the set {0,1,2,...,C}\{0, 1, 2, ..., C\}{0,1,2,...,C}. We develop new heaps incrementally and we will finally see that Dijkstra algorithm can do substantially better than O(m+nlog⁡n)O(m + n \log n)O(m+nlogn).

Time: O(m+nC)O(m + nC)O(m+nC), Space: O(nC)O(nC)O(nC).

As a warm up, we demonstrate an algorithm with runtime O(m+nC)O(m + nC)O(m+nC) and space overhead O(nC)O(nC)O(nC). Here we exclude the space O(m+n)O(m + n)O(m+n) required for storing the graph to simplify the discussion and comparison between the algorithms.

The idea is very simple. We know that the possible distance from any vertex to the source vertex (denoted as sss) is no less than nCnCnC. It suffices to keep an array A1A_1A1​ with length nCnCnC, such that A1[i]A_1[i]A1​[i] maintains the set of vertices with temporary distance iii. In this context, we call A1[i]A_1[i]A1​[i] a bucket and the array A1A_1A1​ itself the buckets.

Initially all A1[i]A_1[i]A1​[i]‘s are empty except that A1[0]A_1[0]A1​[0] contains the source vertex. The algorithm iterates over A1A_1A1​ to find an unmarked vertex with the minimum distance, marked it, update its neighbors’ distances and move them to the corresponding buckets.

Each vertex is inserted into A1A_1A1​ once. There are at most mmm updates of vertices’ distances. As the minimum distance of the unmarked vertices is monotonically increasing, we scan the array A1A_1A1​ only once. Summing over the cost we get the O(m+nC)O(m + nC)O(m+nC) time complexity bound.

In previous section, we successfully reduce the size of A1A_1A1​ to 1+C1 + C1+C. But the runtime remains O(m+nC)O(m + nC)O(m+nC). To motivate the algorithm discussed in this section, we first provide an alternative view of this complexity:

  1. We update the vertices’ distance and move the vertices between buckets, which has cost O(m)O(m)O(m);

  2. At each iteration of Dijkstra’s algorithm, in the worst case, we need to scan the entire array of A1A_1A1​ to find an unmarked vertex with minimum distance, which takes O(C)O(C)O(C) times. This is repeated nnn times.

Where could we have wasted our time? Do we really have spend O(C)O(C)O(C) time to find the unmarked vertex with minimum distance?

We can improve this to O(C)O(\sqrt C)O(C​) by using two-level buckets. We break the buckets in A1A_1A1​ into C+1\sqrt { C + 1}C+1​ blocks, each of size C+1\sqrt { C + 1}C+1​ (assume here C+1\sqrt { C + 1}C+1​ is an integer for simplicity). For each block, we use one flag bit to indicate whether this block is empty. This results in C+1\sqrt { C + 1}C+1​ flags bits, which are stored in bit array A2A_2A2​. From now on,

Now, to find an unmarked vertex with minimum distance, we scan A2A_2A2​ to find the first non-zero bit then the corresponding bucket in A1A_1A1​, which takes O(C)O(\sqrt C)O(C​) time. Totaling the runtime, we obtain a bound of O(m+nC)O(m + n \sqrt C)O(m+nC​).

We are now prepared to analyze the amortized cost of these operations:

  1. Insert: we pay an O(k)O(k)O(k) amortized cost of insertion. Any subsequent pushed down operation on this element is charged to its insertion. Note that an element can be push-down by at most kkk levels.

  2. Decrease-key: we pay an O(1)O(1)O(1) amortized cost of decrease-key operation. The push-down operation is charged to its insertion.

  3. Delete-min: to find the min-element, we scan the bucket block in the lowest level to find the first non-empty bucket. This has time Δ\DeltaΔ. Beginning from the naïve approach, we have improved a lot on both time and space overhead. Amazingly, this is not the end of the story. To motivate what is possibly the inefficient part of O(m+nlog⁡Clog⁡log⁡C)O(m + n \frac{\log C}{\log \log C})O(m+nloglogClogC​) approach, look at the following example.

Suppose that CCC is a large number. Initially the structure is empty. We insert an element v1v_1v1​ followed by v2v_2v2​ and they fall into the same bucket in the kthk^{th}kth level. Next, we perform a delete-min operation. To distinguish, we expand the buckets all the way down to 1st1^{st}1st level.

But since there are two elements, why don’t we compare them directly and return the minimum one? In such case we avoid the unnecessary expansions.

We can extend the idea further. If the number of elements in a bucket is too few, we refuse to expand it. Instead we make of copy of the elements and construct a comparison based priority queue (e.g., Fibonacci heap) on them. Further, note that we are only interested in the possible min-element. Therefore the heap is constructed for only one bucket that contains the min-element.

We use a parameter ttt (to be determined) to control this number. In particular, if the number is at most ttt, we construct a comparison based priority queue. We perform expansion only if the number exceeds ttt.

  1. Insert: insert an element as before. If the element falls into the bucket that is associated with a priority, then also insert this element into the heap.

  2. Decrease-key: if the element is in the bucket that is associated with a priority queue, perform decrease-key in the queue. Otherwise, perform decrease-key as before. If the element falls into the bucket that is associated with a priority queue, then also insert this element into the heap.

  3. Delete-min:

  1. Find the first non-empty bucket in the lowest level
  2. IF the bucket has more than ttt elements THEN
  3. \qquad Expand the bucket
  4. \qquad GO TO STEP 1
  5. ELSE
  6. \qquad IF ∄\nexists∄ queue THEN
  7. \qquad\qquad Construct a queue
  8. \qquad Perform delete-min in the queue
